Setup Guide
Step 1 — Create an API Token

Go to Cloudflare Dashboard → My Profile → API Tokens and create a token with:
• Zone > SSL and Certificates > Edit — to manage custom hostnames
• Account > Workers KV Storage > Edit — to read/write domain records
Scope it to your specific zone.

Step 2 — Get your Zone ID & Account ID

Go to your zone's Overview page in the Cloudflare dashboard.
Both IDs are in the API section on the bottom-right sidebar.

Step 3 — Create the Fallback Origin

In DNS → Records, add:
• Type: AAAA — Name: fallback — Value: 100:: — Proxy: on

Then in SSL/TLS → Custom Hostnames, set the Fallback Origin to fallback.yourdomain.com.

Step 4 — Create the KV Namespace

Run npx wrangler kv namespace create SUBDOMAINS and copy the id into wrangler.toml.
The KV Namespace ID is also visible in Workers & Pages → KV in the dashboard.

KV stores domain records only — writes are minimal (CRUD + cron status changes), so the free plan (1,000 writes/day) handles hundreds of domains fine.
The write limit only matters if you enable KV-based rate limiting. Use Redis instead to bypass it.

Step 5 — Deploy the Worker

Fill in your secrets in .dev.vars, then run:
npm run setup-secrets — pushes secrets to Cloudflare
npm run deploy — validates the token and deploys the worker

Step 6 — Users add their CNAME

Recommended: Ask your user to add the DNS record first (before calling sdk.createSubdomain()).
This speeds up verification and reduces the delay.

The user adds a CNAME record pointing to your fallback origin:
• Type: CNAME — Name: (their chosen subdomain) — Value: fallback.yourdomain.com

Then in your backend, call sdk.createSubdomain('<subdomain>.<user-domain>') with the full hostname the user configured.
The worker's cron job verifies the domain every 5 minutes and sends a webhook when it's active.
